🎯 The Problem

After trying popular meal planning apps like Ollie, Plan to Eat, and Mealime, I encountered a consistent issue:

These apps assume you want to discover and cook hundreds of different recipes.

But here's the reality for many people (including myself):

  • I'm a South Indian who cooks from ~30 family recipes I've grown up with
  • I rotate through these dishes weekly — maybe trying something new once a week
  • I don't need recipe discovery — I need help planning my week from dishes I already know and love
  • My family recipes aren't in any recipe database
  • I want variety within my rotation, not an endless stream of unfamiliar recipes
The existing apps couldn't accommodate this use case.

💡 The Solution

Recipe Vault is purpose-built for rotation-based meal planning from your personal recipe collection.

Core Philosophy

"Most people don't need 10,000 recipes. They need smart planning from the 20–30 dishes they actually cook."

Target Users

  1. Cultural food lovers — people cooking traditional family recipes (South Indian, Korean, Ethiopian, etc.)
  2. Busy professionals — those who prefer a reliable rotation over constant experimentation
  3. Family recipe preservers — documenting recipes passed down through generations
  4. Dietary restriction cooks — those with a limited set of safe, known recipes
  5. Minimalists — people who prefer curation over collection

2. Rotation Intelligence Engine

// Algorithm considers:
// - Last cooked date  → prioritise dishes not made recently
// - Cooking time      → quick meals weekdays, complex on weekends
// - Tag balance       → mix of comfort / light / festive
// - Ingredient overlap → reduce grocery trips
// - Cook count        → slight weight toward personal favourites

public double CalculateRecipeScore(Recipe recipe, DateTime targetDate)
{
    var daysSince = (targetDate - (recipe.LastCookedDate ?? DateTime.MinValue)).Days;
    var recencyScore    = Math.Min(daysSince / 14.0, 1.0);  // max score after 2 weeks
    var popularityScore = recipe.CookCount > 0
        ? Math.Min(recipe.CookCount / 10.0, 0.5)
        : 0;
    var varietyPenalty  = _recentlyPlanned.Contains(recipe.Id) ? -0.5 : 0;

    return recencyScore + popularityScore + varietyPenalty;
}
Strategic Decision: Built a lightweight recommendation engine instead of relying on external APIs or ML models. This keeps the app fast, fully private, and works with ANY cuisine in the world.

3. Smart Weekly Planner

Manual Mode:

  • Drag-and-drop your recipes onto a weekly calendar
  • Visual "last cooked" badges on each recipe card
  • Warning if the same dish is scheduled too frequently

Auto-Suggest Mode:

  • "Surprise me this week" button generates a balanced weekly plan
  • Uses rotation intelligence to ensure variety
  • Respects cooking time preferences per day
// Angular component — drag-drop + auto-generate
@Component({ selector: 'app-weekly-calendar', ... })
export class WeeklyCalendarComponent implements OnInit {
  weekDays = ['Monday', 'Tuesday', 'Wednesday', 'Thursday',
              'Friday', 'Saturday', 'Sunday'];
  mealPlan: { [key: string]: Recipe | null } = {};

  drop(event: CdkDragDrop<Recipe[]>, day: string) {
    if (event.previousContainer !== event.container) {
      this.mealPlan[day] = event.item.data;
      this.saveMealPlan();
    }
  }

  generateWeeklyPlan() {
    this.mealPlanService.generateAutoSuggestions()
      .subscribe(suggestions => {
        suggestions.forEach((recipe, i) => {
          this.mealPlan[this.weekDays[i]] = recipe;
        });
      });
  }
}
Strategic Decision: Provide both manual control AND automation. Power users want control; busy users want a one-click answer.

4. Intelligent Grocery Lists

  • Auto-generated from the current weekly plan
  • Consolidates duplicate ingredients across all recipes
  • Excludes user-defined pantry staples (rice, dal, basic spices)
  • Organized by category for efficient shopping
public async Task<GroceryListDto> GenerateGroceryList(int mealPlanId)
{
    var mealPlan = await _repository.GetMealPlanWithRecipes(mealPlanId);

    var consolidated = mealPlan.Items
        .SelectMany(i => i.Recipe.Ingredients)
        .GroupBy(ing => new { ing.Name, ing.Unit })
        .Select(g => new GroceryItem {
            Name          = g.Key.Name,
            TotalQuantity = g.Sum(i => i.Quantity),
            Unit          = g.Key.Unit
        })
        .Where(i => !i.IsStaple)   // exclude pantry items
        .ToList();

    return new GroceryListDto { Items = consolidated };
}
Strategic Decision: Focus on consolidation and pantry awareness rather than store delivery integrations. Keeps the app simple and works anywhere in the world.

Database Schema

-- Users
CREATE TABLE Users (
    Id            SERIAL PRIMARY KEY,
    Email         VARCHAR(255) UNIQUE NOT NULL,
    PasswordHash  TEXT NOT NULL,
    FirstName     VARCHAR(100),
    CreatedAt     TIMESTAMPTZ DEFAULT NOW()
);

-- Recipes (key fields for rotation tracking)
CREATE TABLE Recipes (
    Id              SERIAL PRIMARY KEY,
    UserId          INT REFERENCES Users(Id),
    Name            VARCHAR(255) NOT NULL,
    LastCookedDate  DATE,          -- core to rotation intelligence
    CookCount       INT DEFAULT 0, -- tracks favourites
    ImageUrl        TEXT,
    CreatedAt       TIMESTAMPTZ DEFAULT NOW()
);

-- Indexes for fast rotation queries
CREATE INDEX idx_recipes_rotation
    ON Recipes(UserId, LastCookedDate);

CREATE TABLE Ingredients (
    Id        SERIAL PRIMARY KEY,
    RecipeId  INT REFERENCES Recipes(Id),
    Name      VARCHAR(255),
    Quantity  DECIMAL,
    Unit      VARCHAR(50),
    IsStaple  BOOLEAN DEFAULT FALSE -- pantry items excluded from grocery list
);

CREATE TABLE MealPlans (
    Id             SERIAL PRIMARY KEY,
    UserId         INT REFERENCES Users(Id),
    WeekStartDate  DATE NOT NULL
);

CREATE TABLE MealPlanItems (
    Id          SERIAL PRIMARY KEY,
    MealPlanId  INT REFERENCES MealPlans(Id),
    RecipeId    INT REFERENCES Recipes(Id),
    DayOfWeek   INT,   -- 0 = Monday … 6 = Sunday
    MealType    VARCHAR(20)  -- breakfast / lunch / dinner
);

🎓 What I Learned

1. Rotation Algorithm Design

Challenge: How to suggest meals that balance variety, cooking time, and user preferences from only 30 recipes — without over-engineering with ML?

Solution: A transparent weighted scoring system.

public double CalculateRecipeScore(Recipe recipe, DateTime targetDate)
{
    var days = (targetDate - (recipe.LastCookedDate ?? DateTime.MinValue)).Days;

    var recencyScore    = Math.Min(days / 14.0, 1.0);
    var popularityScore = recipe.CookCount > 0
                            ? Math.Min(recipe.CookCount / 10.0, 0.5)
                            : 0;
    var varietyPenalty  = _recentlyPlanned.Contains(recipe.Id) ? -0.5 : 0;

    return recencyScore + popularityScore + varietyPenalty;
}
Learning: Simple algorithms can be highly effective. Don't reach for ML until rules genuinely fail.

2. Grocery List Consolidation

Challenge: Ingredients from different recipes use different units (tsp, tbsp, cups, grams). Naively summing them produces nonsense.

Solution: A UnitConverter service normalises to a base unit before summing. Unknown units pass through unchanged rather than crashing.

public class UnitConverter
{
    private readonly Dictionary<string, double> _toMl = new() {
        { "tsp",  5   },
        { "tbsp", 15  },
        { "cup",  240 },
        { "ml",   1   }
    };

    public double ToBase(double qty, string unit) =>
        _toMl.TryGetValue(unit.ToLower(), out var factor)
            ? qty * factor
            : qty; // unknown unit → pass through
}
Learning: Design for graceful degradation. An imperfect grocery list is more useful than a broken one.

3. Image Upload Without Database Bloat

Challenge: Storing recipe images in the database is slow and expensive at scale.

Solution: Angular uploads directly to Azure Blob Storage using short-lived SAS tokens issued by the .NET API. The database stores only the URL. Images are resized to max 1200px / 1 MB on upload.

Learning: Offload specialised tasks to purpose-built services. The database should store data, not files.

4. Angular State Management

Challenge: Meal plan state is needed across multiple components — calendar, recipe list, grocery list — simultaneously.

Solution: A BehaviorSubject in MealPlanService. All subscribed components auto-update without prop-drilling or a heavy library like NgRx.

@Injectable({ providedIn: 'root' })
export class MealPlanService {
  private mealPlan$ = new BehaviorSubject<MealPlan | null>(null);
  plan$ = this.mealPlan$.asObservable();

  update(plan: MealPlan) {
    this.mealPlan$.next(plan);  // all subscribers auto-update
  }
}
Learning: Choose the simplest state tool that solves the problem. RxJS covered all requirements here without NgRx boilerplate.

5. JWT Interceptor for Silent Token Refresh

intercept(req: HttpRequest<any>, next: HttpHandler): Observable<HttpEvent<any>> {
  const token = this.authService.getAccessToken();

  if (token) {
    req = req.clone({
      setHeaders: { Authorization: `Bearer ${token}` }
    });
  }

  return next.handle(req).pipe(
    catchError(err => {
      if (err.status === 401) {
        return this.authService.refreshToken().pipe(
          switchMap(() => next.handle(req))
        );
      }
      return throwError(() => err);
    })
  );
}
Learning: Interceptors are the right place for cross-cutting concerns like auth. Keeps all other services clean.
